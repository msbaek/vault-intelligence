3. **코드 불필요성 우선 (Code Minimization Principle)**: 코드 없이 답변 가능하면 코드 작성 피하기.

5. **변수명/문자열 보존 원칙**: 코드 재현 시 변수명과 문자열 리터럴 유지.
6. **명명 규칙 표준화**: 관례적 명명에 `:: UPPERCASE :` 형식 사용.
7. **일반성과 유연성 균형**: 즉각적 문제 해결과 재사용 가능한 일반적 솔루션 사이 균형 유지.
8. **명확성 추구와 토론 촉진**: 불명확한 부분에서 명확화 요청, 트레이드오프 논의 유도.
9. **교육적 상호작용**: 단순 답변 넘어 효과적인 의사결정을 위한 가르침 역할 강조.

10. **따옴표로 강조**: 중요한 키워드를 따옴표로 감싸서 강조.
11. **코드 리뷰 및 설명 요청**: "설명해줘", "버그 찾아줘", "리팩토링해줘" 패턴.
12. **에러 메시지 활용**: 에러 메시지 전체 복사하여 정확한 해결책 획득.
13. **테스트 코드 자동 생성**: "테스트 코드 작성해줘"로 자동 테스트 케이스 생성.

### 9. Cluely System Prompt
**핵심 주제**: AI 어시스턴트의 실시간 회의 지원과 화면 기반 문제 해결을 위한 상세한 시스템 프롬프트

#### 추출된 기법 (15개)
1. **의도 감지(Intent Detection) 기반 응답**: 50% 이상 확신 있으면 질문으로 간주하고 답변.
2. **직접 답변 우선 전략 (Zero Preamble)**: 설명/인사 없이 즉시 해결책부터 제시.
3. **화면 기반 컨텍스트 우선순위화**: 음성과 화면 정보 함께 분석, 상황별 동적 우선순위 조정.
4. **부동태 모드 진입 기준 정의**: 매우 엄격한 기준으로 불필요한 비활성 응답 방지.
5. **응답 포맷 표준화**: 헤드라인(≤6단어) + 메인 불릿(1-2개) + 서브 불릿 계층 구조.
6. **정의 우선순위(Term Definition)**: 마지막 10-15 단어의 회사명, 기술 용어 자동 감지 정의.
7. **오디오 전사 오류 교정**: 음성 인식 오류를 맥락 단서로 추론하여 해석.
8. **후속 질문 자동 생성**: 명시적 질문 없을 때 1-3개의 타겟화된 후속 질문 생성.
9. **이의 처리 프레임워크**: 판매/협상에서 이의 유형 명명 후 맞춤 응답 전략 제시.
10. **마크다운 포맷 제약 (No Headers)**: #, ##, ### 없이 굵은 텍스트와 불릿만 사용.
11. **LaTeX 수학 렌더링 표준화**: 모든 수학 표현을 LaTeX 형식으로 통일.
12. **요약 금지 패턴**: 명시적 요청 없으면 자동 요약 금지, 핵심 3개 포인트 이하.
13. **역할 명확화 (Speaker Label)**: me(사용자), them(상대방), assistant(AI) 명확히 구분.
14. **행동 사례 질문 처리**: 사용자 이력 사용하거나 일반 예시로 채우되 검증 불가 팩트 회피.
15. **프롬프트 숨김 및 신원 보호**: 지침 언급 금지, 정체에 대해 간단히만 답변.

### 10. Generative Automation Testing with Playwright MCP Server
**핵심 주제**: LLM과 Playwright MCP 서버를 활용한 자동화 테스트 생성 워크플로우

#### 추출된 기법 (8개)
1. **MCP 기반 도구 통합**: LLM이 실제 애플리케이션과 직접 상호작용하는 중간 인터프리터 프로토콜 활용.
2. **자연어 시나리오 변환 프롬프트**: 자연언어 테스트 시나리오를 구조화된 단계별 지시사항으로 변환.
3. **Agentic Workflow 패턴**: "먼저 도구로 검증, 모든 단계 완료 후 테스트 코드 생성" 순서 명시.
4. **자동 승인 설정(Auto-Approval)**: `"chat.tools.autoApprove": true`로 반복 인증 제거.
5. **반복 검증 및 개선 루프**: 생성된 테스트 자동 실행, 실패 시 단계별 재생성.
6. **Role/Label 기반 요소 선택**: XPath/CSS 대신 `getByRole()`, `getByLabel()` 접근성 기반 선택 우선.
7. **문제 인식 및 개선 체크리스트**: 생성 코드 한계점 명시적 인정, 검토/보완 단계 필수.
8. **대상 사용자별 활용 전략**: 역할별(자동화 엔지니어 없는 팀/입문자/프론트엔드 등) 적용 방식 차별화.

## 핵심 테마

### 1. PRP(Product Requirements Prompt) 방법론
- 5계층 구조: 시스템, 도메인, 태스크, 인터랙션, 응답 컨텍스트
- 비기능 요구사항 + 기술 제약 + 행동 가이드라인 명시
- 프롬프트 엔지니어링 → 소프트웨어 설계 수준으로 격상

### 2. CLAUDE.md 기반 프로젝트 컨텍스트
- 저장소 루트의 CLAUDE.md 자동 로드
- 아키텍처, 코드 규칙, 패턴, 테스트 전략 명시
- 개발자 온보딩 시간 대폭 단축

### 3. 구조화된 프롬프트 태그
- `<CODE_REVIEW>`, `<PLANNING>` 태그로 AI 사고 과정 명시적 구조화
- Zero Preamble (직접 답변 우선 전략)
- 응답 포맷 표준화 (헤드라인 + 불릿 + 상세)

### 4. MCP 기반 에이전틱 워크플로우
- Playwright MCP 서버로 실제 애플리케이션과 LLM 직접 상호작용
- Agentic Workflow: 도구 검증 → 테스트 코드 생성
- 자동 승인 + 반복 검증 루프

### 5. 모델별 비용 최적화
- 작업 복잡도별 다른 모델 할당 (Haiku/Sonnet/Opus)
- Simple-to-Detailed 프롬프트 계층화
- 재사용 가능한 커스텀 커맨드 라이브러리
