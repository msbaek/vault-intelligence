## 프롬프트 작성 5원칙

### 1. **구체적으로 명시하라 (Be Specific)**
```
❌ "코드를 개선해줘"
✅ "이 함수의 시간 복잡도를 O(n²)에서 O(n log n)으로 개선해줘"
```

### 2. **컨텍스트를 제공하라 (Provide Context)**
```
❌ "버그를 수정해줘"
✅ "사용자가 빈 문자열을 입력하면 NullPointerException이 발생해.
   입력 검증을 추가해서 수정해줘"
```

### 3. **제약 조건을 명시하라 (Set Constraints)**
```
❌ "테스트를 작성해줘"
✅ "기존 테스트 스타일을 유지하면서, 엣지 케이스 3개를 포함한
   단위 테스트를 작성해줘. 모킹은 사용하지 마"
```

### 4. **출력 형식을 지정하라 (Define Output)**
```
❌ "API 문서를 만들어줘"
✅ "OpenAPI 3.0 형식으로 API 문서를 작성해줘.
   각 엔드포인트에 예시 요청/응답을 포함해"
```

### 5. **역할을 부여하라 (Assign Role)**
```
❌ "코드 리뷰해줘"
✅ "시니어 백엔드 개발자로서 이 코드를 리뷰해줘.
   보안, 성능, 유지보수성 관점에서 평가해"
```

## 필수 파일 구조

### CLAUDE.md (프로젝트 루트)
```markdown
# 프로젝트 가이드

## 아키텍처
- Clean Architecture / Hexagonal
- 레이어: Domain → Application → Infrastructure

## 코딩 규칙
- 함수는 20줄 이내
- 단일 책임 원칙 준수
- 테스트 커버리지 80% 이상

## 명령어
- `npm run test` - 테스트 실행
- `npm run lint` - 린트 검사
```

### AGENTS.md (서브에이전트 정의)
```yaml
code-reviewer:
  role: 코드 품질 검토 전문가
  when: PR 생성 시 자동 실행
  tools: Read, Grep

test-writer:
  role: TDD 전문가
  when: 새 기능 구현 요청 시
  tools: Read, Write, Bash
```

## 핵심 프롬프트 템플릿

### 1. STICC 템플릿
```markdown
**Situation**: [현재 상황 설명]
**Task**: [수행할 작업]
**Intention**: [의도와 목표]
**Concern**: [우려사항, 제약조건]
**Calibration**: [확인 요청, 질문]
```

### 2. TDD 구현 요청
```markdown
다음 테스트가 통과하도록 최소한의 구현을 해줘:
[테스트 코드]

규칙:
- 테스트를 수정하지 마
- 최소한의 코드만 작성해
- 하드코딩으로 시작해도 좋아
```

### 3. 코드 리뷰 요청
```markdown
시니어 개발자로서 이 PR을 리뷰해줘:
[코드 또는 diff]

다음 관점에서 평가해:
1. 보안 취약점
2. 성능 이슈
3. 유지보수성
4. 테스트 커버리지

각 이슈에 대해 우선순위(Critical/Major/Minor)를 표시해
```

### 4. 디버깅 요청
```markdown
**에러 메시지**: [에러 내용]
**재현 단계**: [1, 2, 3...]
**기대 동작**: [예상 결과]
**실제 동작**: [실제 결과]

원인을 분석하고 해결 방안을 제시해줘.
불확실한 부분이 있으면 먼저 물어봐.
```

### 5. 리팩토링 요청
```markdown
이 코드를 리팩토링해줘:
[코드]

목표:
- [가독성 향상 / 성능 개선 / 모듈화 등]

제약:
- 기존 테스트가 모두 통과해야 해
- 공개 API는 변경하지 마
```

## TDD + AI 빠른 사이클

```
┌─────────────────────────────────────────┐
│  1. Red (개발자)                         │
│     - 실패하는 테스트 작성                 │
│     - 의도를 명확히 표현                   │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  2. Green (AI)                          │
│     - "이 테스트만 통과시켜줘"             │
│     - 최소한의 구현                       │
└─────────────────┬───────────────────────┘
                  ▼
┌─────────────────────────────────────────┐
│  3. Refactor (AI + 개발자)               │
│     - "리팩토링 제안해줘"                  │
│     - 개발자가 검토 후 승인                │
└─────────────────────────────────────────┘
```

|--------|------|

## AI 활용 안전 수칙

1. **AI 출력물을 맹신하지 마라** - 항상 검토
2. **테스트 없이 배포하지 마라** - AI 코드도 테스트 필수
3. **민감 정보를 전달하지 마라** - API 키, 비밀번호 주의
4. **롤백 전략을 준비하라** - Git 활용
5. **시니어 검토를 받아라** - 중요한 변경은 필수
