## 문서 61: Building AI Agents using Java and Embabel

### 추출된 기법들

1. **타입 기반 워크플로우 제어 (Type-based Workflow Control)**
   - 설명: 각 AI 에이전트 단계의 입력/출력을 명확한 타입으로 정의하여, AI가 올바른 타입으로 데이터를 변환할 수 없으면 다음 단계로 진행하지 않도록 제어
   - 카테고리: AI 에이전트
   - 실무 적용: AI 파이프라인 설계 시 각 단계를 DTO/Record로 명확히 정의하고, 타입 검증을 통해 잘못된 출력이 전파되는 것을 방지

2. **GOAP (Goal Oriented Action Planning) 플래닝**
   - 설명: 게임 개발에서 검증된 결정론적 플래닝 시스템을 AI 에이전트에 적용. 함수들만 정의하면 플래너가 자동으로 실행 그래프를 생성하고, 각 단계 완료 후 재계획 수행
   - 카테고리: AI 에이전트
   - 실무 적용: AI 에이전트 설계 시 명시적 워크플로우 대신 입력/출력 타입만 정의하고 플래너가 실행 순서를 결정하도록 위임

3. **모델 메타데이터 기반 동적 선택**
   - 설명: 각 LLM의 메타데이터(가격, 지식 컷오프, 토큰 비용 등)를 설정하여 작업 특성에 맞는 최적 모델을 동적으로 선택
   - 카테고리: 워크플로우
   - 실무 적용: 역사적 질문 → 저렴한 모델, 최신 정보 필요 → 최신 지식 컷오프 모델, Fallback 전략 활용

4. **도메인 주도 설계(DDD)와 AI 에이전트 결합**
   - 설명: DDD 원칙을 AI 에이전트 설계에 적용하여 워크플로우를 도메인 타입으로 모델링
   - 카테고리: AI 에이전트
   - 실무 적용: 비즈니스 도메인 객체를 AI 파이프라인의 핵심 타입으로 사용

5. **관측 가능성(Observability) 내장 설계**
   - 설명: 각 단계별 실행 로그, 토큰 사용량, 성능 메트릭을 상세히 추적
   - 카테고리: 워크플로우
   - 실무 적용: AI 에이전트 개발 시 처음부터 로깅/메트릭 수집을 설계에 포함

6. **병렬 처리를 통한 성능 최적화**
   - 설명: Spring의 병렬 스트림 기능을 활용하여 독립적인 AI 작업을 병렬로 수행
   - 카테고리: 워크플로우
   - 실무 적용: 여러 문서 분석, 다중 프롬프트 실행 등 독립적 작업은 `parallelStream()` 활용

7. **@AchievesGoal 어노테이션을 통한 목표 명시**
   - 설명: 최종 목표를 달성하는 함수에 어노테이션으로 목표를 명시하여 플래너가 역방향으로 계획 수립
   - 카테고리: AI 에이전트
   - 실무 적용: 복잡한 에이전트에서 최종 목표를 명확히 선언

## 문서 62: Can Junie be a real competitor for Cursor, Windsurf, and VS Code Copilot?

### 추출된 기법들

1. **Guidelines 파일을 통한 AI 에이전트 개인화**
   - 설명: 루트 디렉토리에 `.junie/guidelines.md` 파일을 정의하여 AI 에이전트의 동작을 프로젝트 맞춤형으로 설정
   - 카테고리: AI 에이전트
   - 실무 적용: CLAUDE.md, Cursor Rules 등 각 AI 도구에 맞는 가이드라인 파일 작성

2. **Code/Ask 모드 분리 활용**
   - 설명: 코드 생성(Code)과 질문(Ask) 모드를 목적에 맞게 구분하여 사용
   - 카테고리: 워크플로우
   - 실무 적용: 코드 작업 시 Code 모드, 아키텍처 조언 시 Ask 모드 사용

3. **점진적 프롬프트와 후속 작업(Follow-up) 패턴**
   - 설명: 한 번에 모든 요구사항을 주는 대신, 후속 작업 기능을 사용하여 점진적으로 요구사항을 명확히 함
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: 복잡한 기능은 단계별로 나누어 요청

4. **AI 생성 코드의 품질 검증 체크리스트**
   - 설명: AI가 생성한 코드에서 흔히 누락되는 항목들을 체크
   - 카테고리: 코드 품질
   - 실무 적용: Docker 설정, 테스트 코드, 통합 테스트, 라이브러리 통합 확인

5. **계획(Plan) 기반 작업 수행 검토**
   - 설명: AI 에이전트가 프롬프트를 받으면 먼저 계획을 수립하고, 이 계획을 사용자가 검토
   - 카테고리: AI 에이전트
   - 실무 적용: AI가 생성한 계획을 먼저 검토한 후 실행 승인

6. **자동 승인 vs 단계별 승인 전략**
   - 설명: Brave Mode (자동 승인)와 단계별 승인을 상황에 맞게 선택
   - 카테고리: 워크플로우
   - 실무 적용: 간단한 작업은 Brave Mode, 중요한 작업은 단계별 승인

7. **구체적인 기술 스택 명시**
   - 설명: 프롬프트에 사용할 기술/라이브러리를 명시적으로 지정
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: "JdbcClient를 사용하세요", "Vaadin Flow를 사용하여" 등 구체적 지정

## 문서 64: ChatGPT Prompt Engineering for Developers

### 추출된 기법들

1. **명확하고 구체적인 지시 작성**
   - 설명: 프롬프트가 길더라도 명확한 지시를 제공하는 것이 더 좋은 결과를 가져옴
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: 원하는 결과의 형식, 스타일, 제약조건을 명시적으로 기술

2. **구분자(Delimiter) 사용**
   - 설명: 입력의 서로 다른 부분을 ```, """, <>, `<tag></tag>` 등으로 구분
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: 코드 블록, 요구사항, 컨텍스트 등을 구분

3. **구조화된 출력 요청**
   - 설명: JSON, HTML 등 특정 형식으로 출력을 요청
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: API 응답, 데이터 변환, 설정 파일 생성 시 정확한 구조 명시

4. **조건 검증 요청**
   - 설명: 모델에게 입력이 특정 조건을 만족하는지 먼저 확인하도록 지시
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: "절차가 있으면 정리하고, 없으면 'No steps provided'라고 응답"

5. **Few-shot 프롬프팅**
   - 설명: 원하는 응답 스타일의 예시를 먼저 제공
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: 코드 리뷰 코멘트 스타일, 커밋 메시지 형식 등을 예시로 제공

6. **모델에게 생각할 시간 주기**
   - 설명: 복잡한 문제 해결 시 중간 단계를 명시적으로 요청
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: "먼저 분석하고, 그 다음 설계하고, 마지막으로 구현하세요"

7. **단계별 작업 지정**
   - 설명: 작업을 1-2-3-4 순서의 명확한 단계로 분해하여 지시
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: 복잡한 작업을 순차적 단계로 분해

8. **출력 형식 명시**
   - 설명: 원하는 출력 형식을 템플릿으로 제공
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: 코드 리뷰, 버그 분석 보고서 등의 일관된 형식 유지

9. **스스로 해결 후 비교하기**
   - 설명: 검증 작업 시 모델에게 먼저 자체 솔루션을 도출한 후 비교하도록 지시
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: "먼저 어떻게 구현할지 생각한 후, 현재 구현과 비교하여 개선점을 찾아라"

10. **Temperature 조절**
    - 설명: temperature=0은 결정론적, 높은 값은 창의적 출력
    - 카테고리: 프롬프트 엔지니어링
    - 실무 적용: 코드 생성은 temperature=0, 아이디어 생성은 높은 temperature

11. **반복적 프롬프트 개발**
    - 설명: 첫 프롬프트에서 완벽한 결과를 기대하지 않고, 지속적으로 개선
    - 카테고리: 워크플로우
    - 실무 적용: 프롬프트 버전 관리, A/B 테스트

   - 실무 적용: architect agent에게는 읽기 권한만 부여

   - 카테고리: AI 에이전트
   - 실무 적용: 에이전트 설명(prompt)을 명확히 정의

   - 카테고리: 워크플로우
   - 실무 적용: `.claude/agents/` 디렉토리에 프로젝트별 설정 저장

## 문서 70: Claude Code Tip - Land then Expand

### 추출된 기법들

1. **Land then Expand (착지 후 확장) 전략**
   - 설명: 대규모 PRD로 시작하지 않고, 최소한의 핵심 기능으로 작동하는 기반을 먼저 구축한 후 점진적으로 확장
   - 카테고리: 워크플로우
   - 실무 적용: 3-4개의 핵심 필드만으로 DB 테이블, API, CRUD 먼저 생성

2. **Thinking 모델의 컨텍스트 관리**
   - 설명: thinking 모델에서는 큰 문서가 오히려 혼란을 야기하므로, 초기에는 최소 컨텍스트로 시작
   - 카테고리: 프롬프트 엔지니어링
   - 실무 적용: 첫 프롬프트에서는 핵심 요구사항만 전달

3. **패턴 기반 확장 (Expand with Context)**
   - 설명: 작동하는 기반이 있으면 AI가 기존 패턴을 따라 향상시킴
   - 카테고리: AI 에이전트
   - 실무 적용: 기본 CRUD가 완성된 후, 실제 파일을 컨텍스트로 제공하면서 확장 요청

4. **4단계 실용적 워크플로우**
   - 설명: Land → Expand → Iterative expansion → Polish
   - 카테고리: 워크플로우
   - 실무 적용: 최소 버전 → 컨텍스트 전달 → 기능별 반복 확장 → 스타일링/오류 처리

5. **Walking Skeleton 전략과의 연계**
   - 설명: 전체 아키텍처를 관통하는 최소한의 end-to-end 동작을 먼저 구현
   - 카테고리: TDD
   - 실무 적용: 3가지 핵심 필드만으로 시작

6. **짧은 반복 루프 우선**
   - 설명: 큰 요청보다 짧은 반복 루프가 현재 AI 코딩에서 더 효과적
   - 카테고리: 생산성
   - 실무 적용: 작은 단위로 나누어 요청, 각 단계에서 결과 확인 후 다음 단계 진행
